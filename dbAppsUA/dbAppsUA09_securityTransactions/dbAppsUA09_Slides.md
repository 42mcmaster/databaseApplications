# Безпека та Транзакції

## Захист ваших даних

**Що ви навчитесь:**
- Як запобігти SQL-атакам
- Як виконувати безпечні операції з базою даних

---

# Частина 1: SQL-ін'єкція

## Нормальний вхід

Користувач натискає кнопку входу. Система перевіряє облікові дані:

```python
username = "alice"
password = "secret123"

query = "SELECT * FROM users WHERE username = '" + username + "' AND password = '" + password + "'"
# Результат: SELECT * FROM users WHERE username = 'alice' AND password = 'secret123'
```

Якщо облікові дані совпадають — вхід дозволено. ✓

---

## Атака SQL-ін'єкція

Але що якщо зловмисник введе щось хитре?

```python
username = "admin' OR '1'='1"
password = "anything"

# Небезпечний код
query = "SELECT * FROM users WHERE username = '" + username + "' AND password = '" + password + "'"
# Результат: SELECT * FROM users WHERE username = 'admin' OR '1'='1' AND password = 'anything'
```

Де проблема? `'1'='1'` завжди істинно! Запит повертає **всіх користувачів**.

---

## Чому це страшно?

SQL-ін'єкція дозволяє зловмиснику:
- Читати приватні дані (паролі, номери телефонів)
- Змінювати або видаляти записи
- Обходити перевірку паролів
- Взяти під контроль весь сервер

**Це одна з найнебезпечніших вразливостей веб-додатків.**

---

## Рішення: Параметризовані запити

Замість конкатенації — використовуйте заповнювачі (placeholders):

```python
# БЕЗПЕЧНО
username = "admin' OR '1'='1"
password = "anything"

query = "SELECT * FROM users WHERE username = ? AND password = ?"
cursor.execute(query, (username, password))
```

Система трактує введене значення як **дані**, а не код. SQL-структура захищена.

---

## Небезпечно vs Безпечно (в одному слайді)

| Небезпечно | Безпечно |
|-----------|---------|
| `query = "... WHERE id = " + str(user_id)` | `query = "... WHERE id = ?"` + `(user_id,)` |
| Конкатенація строк | Параметризовані запити |
| Введення користувача як SQL-код | Введення як дані |
| **НІКОЛИ не робіть так!** | ✓ Завжди так робіть |

---

## Золоте правило безпеки

> **Ніколи** не конкатенуйте введення користувача прямо в SQL-запити.
> **Завжди** використовуйте параметризовані запити.

Це найпростіший спосіб уникнути SQL-ін'єкції.

# Частина 2: Транзакції

## Проблема: Переведення грошей в банку

Уявіть: Алісе потрібно перевести \$100 Бобу.

**Кроки:**
1. Зніміть \$100 з рахунку Алісі
2. Додайте \$100 на рахунок Боба

**Що якщо:**
- Крок 1 виконаний ✓
- Система падає перед кроком 2 ✗

Результат: Алісе знято \$100, але Боб не отримав нічого!

---

## Що таке транзакція?

Транзакція (transaction) — це **послідовність операцій**, яка повинна виконуватись **повністю** або **зовсім не виконуватись**.

Для переведення грошей:
- Либо обидва кроки успішні ✓
- Либо обидва скасовуються (ROLLBACK) ↺

**Середнього нема!**

---

## SQL синтаксис транзакцій

```sql
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE user_id = 1;
UPDATE accounts SET balance = balance + 100 WHERE user_id = 2;
COMMIT;
```

| Команда | Значення |
|---------|----------|
| `BEGIN;` | Початок транзакції |
| `COMMIT;` | Зберегти всі зміни |
| `ROLLBACK;` | Скасувати всі зміни |

---

## ROLLBACK скасовує зміни

```sql
BEGIN;
UPDATE accounts SET balance = 0 WHERE user_id = 1;
-- Ой! Помилка!
ROLLBACK;
-- Баланс повернувся до попереднього значення ✓
```

Якщо щось піде не так, просто викличте `ROLLBACK;` — усі зміни будуть скасовані.

---

## Коли використовувати транзакції?

Використовуйте транзакції коли:
- ✓ Операції залежать одна від одної (переведення грошей)
- ✓ Усі кроки мають виконуватись разом
- ✓ Потрібна безпека від помилок

Не потрібні для:
- ✗ Простих вставок одного рядка
- ✗ Незалежних операцій

---

## ACID властивості

Хорошие транзакції мають чотири властивості:

| Властивість | Значення |
|------------|----------|
| **A**tomicity | Атомарність — все або нічого |
| **C**onsistency | Узгодженість — дані залишаються коректними |
| **I**solation | Ізоляція — транзакції не мешають одна одній |
| **D**urability | Надійність — при COMMIT дані збережені назавжди |
