# Будування та управління даними
## CREATE, INSERT, UPDATE, DELETE та обмеження

**Розробка застосунків баз даних**
Інженерія програмного забезпечення | Мединський центр кар'єри

---

## Короткий повтор

До цього ми **читали** дані з існуючих баз даних:
- SELECT, WHERE, JOINs, GROUP BY…

Сьогодні ми робимо навпаки і **будуємо свою** базу даних з нуля.

Ви навчитеся:
1. **Створювати** таблицю
2. **Додавати** рядки в неї
3. **Оновлювати** існуючі рядки
4. **Видаляти** ненотрібні рядки
5. **Захищати** таблицю обмеженнями (Constraints), щоб погані дані не потрапили туди

---

## Чотири основні операції

| Операція | Що вона робить |
|----------|-----------|
| **CREATE TABLE** | Будує нову порожню таблицю зі стовпцями та правилами |
| **INSERT** | Додає нові рядки |
| **UPDATE** | Змінює існуючі рядки |
| **DELETE** | Видаляє рядки |

SELECT запитує "що там лежить?" — ці чотири операції насправді змінюють, що там лежить.

---

## CREATE TABLE — основи

Ви робите план. Ви повідомляєте SQL:
- Як називається таблиця
- Які стовпці в ній є
- Якого типу дані в кожному стовпці

```sql
CREATE TABLE students (
  student_id  INTEGER,
  name        TEXT,
  gpa         REAL
);
```

Готово. Порожня таблиця, готова для рядків.

---

## Типи даних SQLite (спрощена версія)

| Тип | Використовується для |
|------|-----------|
| **INTEGER** | Цілі числа (id, вік, кількість) |
| **TEXT** | Текст — імена, e-mail, описи |
| **REAL** | Числа з крапкою (gpa, ціна, зріст) |
| **BLOB** | Двійкові дані — зображення, файли (рідко в класі) |
| **NULL** | Стовпець ще не має значення |

Вибирайте найменший тип, який підходить. Якщо ви ніколи не зберігатимете десяткові числа, використовуйте INTEGER, а не REAL.

---

## INSERT INTO — додавання рядків

```sql
INSERT INTO students (student_id, name, gpa)
VALUES (1, 'Arnold', 3.85);
```

**Читайте як речення:**
> "INSERT INTO (додай в) students (студентів), використовуючи ці стовпці, ці значення."

- Текстові значення потребують **лапки**: `'Arnold'`
- Числа ні: `3.85`
- Порядок значень має відповідати порядку стовпців, які ви вказали.

---

## INSERT — декілька рядків одночасно

Ви можете складати рядки в одній операції:

```sql
INSERT INTO students (student_id, name, gpa)
VALUES
  (1, 'Arnold',      3.85),
  (2, 'Lebron',      3.70),
  (3, 'Kobe',        3.92),
  (4, 'Maxibillion', 2.45);
```

Швидко, чисто та легко читати.

---

## UPDATE — зміна існуючих рядків

```sql
UPDATE students
SET gpa = 3.90
WHERE student_id = 1;
```

**Читайте як речення:**
> "UPDATE (оновити) students таблицю, SET (встановити) gpa на 3.90, WHERE (де) student_id дорівнює 1."

Умова `WHERE` говорить SQL **які рядки** змінювати.

---

## ⚠️ Помилка номер 1 у новачків

Що буде, якщо ви забудете WHERE?

```sql
UPDATE students SET gpa = 4.0;
```

**Кожен студент має тепер 4.0.**
Без попередження. Без скасування. Ви просто переписали всю таблицю.

Це найпоширеніша та найболючіша помилка в SQL. Будьте обережні.

---

## Безпечний робочий процес UPDATE

Завжди робіть ЦЕ В ДВА КРОКИ:

**Крок 1 — перевірте WHERE за допомогою SELECT:**
```sql
SELECT * FROM students WHERE gpa < 2.0;
```
Подивіться на рядки. Це ті, які ви насправді хочете змінити?

**Крок 2 — тепер UPDATE:**
```sql
UPDATE students SET status = 'probation' WHERE gpa < 2.0;
```

**Спочатку SELECT, потім UPDATE.** Зробіть це звичкою.

---

## DELETE — видалення рядків

```sql
DELETE FROM students
WHERE student_id = 4;
```

Та ж структура як UPDATE: вам потрібен WHERE, щоб сказати **які** рядки.

І та ж страшна пастка…

```sql
DELETE FROM students;
```

**Це видаляє КОЖЕН рядок в таблиці.** Готово. Просто так.

---

## Безпечний робочий процес DELETE

Та ж звичка в два кроки:

```sql
-- Крок 1: дивіться, що ви видалите
SELECT * FROM students WHERE student_id = 4;

-- Крок 2: видаліть тільки, якщо результати виглядають добре
DELETE FROM students WHERE student_id = 4;
```

> **"Спочатку SELECT, потім UPDATE або DELETE. Завжди."**

## Чому обмеження (Constraints)?

Поки що наш CREATE TABLE був відкритий. Кожен міг:
- Залишити ім'я пусте
- Додати двох студентів з одним і тим ж ID
- Ввести від'ємну GPA
- Вигадати `dept_id`, який не існує

**Обмеження (Constraints)** — це правила, які ви вбудовуєте в таблицю, щоб погані дані не потрапили туди.

> **Погані дані на вході → погані дані на виході.** Обмеження — це як ви тримаєте сміття подалі.

---

## Обмеження 1 & 2: NOT NULL та UNIQUE

**NOT NULL** — "Цей стовпець повинен мати значення"
```sql
CREATE TABLE students (
  student_id INTEGER,
  name       TEXT NOT NULL
);
```

**UNIQUE** — "Два рядки не можуть мати одне й те ж значення в цьому стовпці"
```sql
CREATE TABLE users (
  user_id INTEGER,
  email   TEXT UNIQUE
);
```

NOT NULL: ви не можете додати студента без імені. UNIQUE: ви не можете додати двох користувачів з одним e-mail. Ідеально для e-mailів, імен користувачів, номерів телефонів.

---

## Обмеження 3 & 4: DEFAULT та CHECK

**DEFAULT** — "Якщо ніхто не передасть значення, використайте це"
```sql
CREATE TABLE posts (
  post_id    INTEGER,
  created_at TEXT DEFAULT CURRENT_TIMESTAMP
);
```

**CHECK** — "Значення повинно зробити цю умову істинною"
```sql
CREATE TABLE products (
  product_id INTEGER,
  price      REAL CHECK (price > 0)
);
```

DEFAULT: якщо ви додасте запис без `created_at`, SQL заповнить поточний час. CHECK: спробуйте додати продукт з `price = -5`? Відхилено.

---

## Обмеження 5: PRIMARY KEY

Велике. Це означає:
- Цей стовпець — **унікальний ID** для кожного рядка
- Значення повинні бути унікальні (UNIQUE)
- Значення не можуть бути пусті (NOT NULL)
- SQL зробить пошук по цьому стовпцю дуже швидким

```sql
CREATE TABLE courses (
  course_id INTEGER PRIMARY KEY,
  title     TEXT NOT NULL
);
```

Кожна хороша таблиця повинна мати первинний ключ (Primary Key).

---

## Обмеження 6: FOREIGN KEY

"Цей стовпець повинен відповідати первинному ключу в іншій таблиці."

```sql
CREATE TABLE enrollments (
  enrollment_id INTEGER PRIMARY KEY,
  student_id    INTEGER REFERENCES students(student_id),
  course_id     INTEGER REFERENCES courses(course_id)
);
```

Тепер ви не можете записати студента, який не існує, або в курс, який не існує. Це називається **референціальна цілісність (Referential integrity)** — вашіFK завжди вказують на щось реальне.

---

## Короткий огляд: що блокує кожне обмеження?

| Правило | Блокує… |
|------|---------|
| NOT NULL | Пусті значення |
| UNIQUE | Дублікати значень |
| DEFAULT | Забуті значення (заповнює їх) |
| CHECK | Невалідні / неможливі значення |
| PRIMARY KEY | Пусті ID та дублікати ID |
| FOREIGN KEY | Посилання на рядки, які не існують |

Кожне блокує іншу категорію "поганих даних".

---

## Весь рецепт

1. **CREATE TABLE** зі стовпцями, типами та обмеженнями.
2. **INSERT** рядки, використовуючи список стовпців + VALUES.
3. Щоб змінити дані, **спочатку SELECT**, щоб перевірити → потім **UPDATE**.
4. Щоб видалити дані, **спочатку SELECT**, щоб перевірити → потім **DELETE**.
5. Використовуйте **обмеження**, щоб майбутня вам (та майбутні студенти) не могли зламати таблицю.

---

## Основні висновки

- SELECT читає дані. CREATE / INSERT / UPDATE / DELETE змінюють їх.
- **Завжди SELECT перед UPDATE або DELETE.** Завжди.
- Обмеження — це не додаткова робота — це як ви тримаєте дані чистими.
- PRIMARY KEY + FOREIGN KEY — це то, що перетворює окремі таблиці на реальну реляційну базу даних.
