## Словник та ключові поняття

### Основні терміни баз даних

1. **Сутність (Entity)** — людина, місце, річ або подія, про яку ви хочете зберігати дані. Сутності стають таблицями.

2. **Атрибут (Attribute)** — властивість сутності. Атрибути стають стовпцями в таблиці.

3. **Таблиця (Table)** — впорядкована колекція даних, організована в рядки та стовпці. Кожна таблиця представляє одну сутність.

4. **Рядок (Row/Record)** — один екземпляр сутності; один повний запис в таблиці.

5. **Стовпець (Column/Field)** — названий атрибут, який з'являється в кожному рядку.

6. **Первинний ключ (Primary Key, PK)** — стовпець (або набір стовпців), який унікально ідентифікує кожний рядок. PK не можуть бути NULL і не можуть повторюватися.

7. **Іноземний ключ (Foreign Key, FK)** — стовпець, який посилається на первинний ключ іншої таблиці. FK створюють зв'язки між таблицями.

8. **Композитний ключ (Composite Key)** — первинний ключ, складений з двох або більше стовпців. Використовується, коли жоден одиночний стовпець не є унікальним.

9. **Зв'язок (Relationship)** — посилання між таблицями, визначена первинними та іноземними ключами.

10. **Один-до-Багатьох (One-to-Many, 1:M)** — один рядок в Таблиці A пов'язаний з багатьма рядками в Таблиці B, але кожен рядок B пов'язаний тільки з одним рядком A. (Приклад: один замовник → багато замовлень.)

11. **Багато-до-Багатьох (Many-to-Many, M:M)** — рядки обох таблиць можуть пов'язуватися з багатьма рядками в іншій. Потребує з'єднавальної таблиці. (Приклад: студенти ↔ класи.)

12. **З'єднавальна таблиця (Junction Table)** — посередницька таблиця, яка з'єднує дві таблиці в M:M зв'язку. Містить FK для обох.

### Терміни якості даних

13. **Надмірність (Redundancy)** — зберігання однієї й тієї ж інформації кілька разів. Витрачає місце та створює проблеми з оновленням.

14. **Аномалія оновлення (Update Anomaly)** — невідповідні дані, викликані оновленням однієї копії значення та забуванням інших.

15. **Аномалія вставлення (Insert Anomaly)** — неможливість додати нові дані без дублювання іншої інформації.

16. **Аномалія видалення (Delete Anomaly)** — видалення одного фрагмента даних випадково стирає пов'язані дані.

17. **Цілісність даних (Data Integrity)** — точність та узгодженість даних у базі даних. Хороший дизайн її захищає; надмірність її ламає.

## Приклади SQL

Це SQL шаблони, які вам потрібно знати для цього модуля.

### SELECT — витяг даних з однієї таблиці

```sql
SELECT primaryTitle, startYear
FROM title_basics
WHERE startYear = 2024;
```

### WHERE з AND / OR

```sql
SELECT *
FROM title_basics
WHERE titleType = 'movie'
  AND startYear >= 2020;
```

### LIKE з підстановочними символами

Символ `%` означає "будь-які символи."

```sql
WHERE primaryTitle LIKE '%Avengers%'   -- містить "Avengers"
WHERE primaryTitle LIKE 'The%'         -- починається з "The"
```

### IN — скорочення для кількох умов OR

```sql
WHERE category IN ('actor', 'actress')
-- те саме що: WHERE category = 'actor' OR category = 'actress'
```

### IS NULL / IS NOT NULL

Фільтруйте рядки, де значення є (або не є) відсутнім. Ви **повинні** використовувати `IS NULL`, ніколи `= NULL`.

```sql
WHERE runtimeMinutes IS NOT NULL
```

### JOIN — об'єднання двох таблиць

```sql
SELECT b.primaryTitle, r.averageRating
FROM title_basics b
JOIN title_ratings r ON b.tconst = r.tconst
WHERE r.averageRating >= 8.0;
```

Clausula `ON` розповідає SQL, які стовпці збігати. `b` та `r` — це псевдоніми таблиць — скорочення, щоб ви не переписували повну назву таблиці.

### JOIN через три таблиці

```sql
SELECT b.primaryTitle, n.primaryName, p.category
FROM title_basics b
JOIN title_principals p ON b.tconst = p.tconst
JOIN name_basics n ON p.nconst = n.nconst
WHERE b.primaryTitle = 'Inception';
```

### Функції агрегування — COUNT, AVG, SUM, MIN, MAX

```sql
SELECT COUNT(*) FROM title_basics;
SELECT AVG(averageRating) FROM title_ratings;
SELECT MAX(startYear) FROM title_basics;
```

### COUNT(DISTINCT стовпець)

Рахує тільки унікальні значення.

```sql
SELECT COUNT(DISTINCT tconst) FROM title_principals;
```

### GROUP BY

Групуйте рядки, що мають спільне значення, та агрегуйте кожну групу.

```sql
SELECT startYear, COUNT(*) AS movie_count
FROM title_basics
WHERE titleType = 'movie'
GROUP BY startYear
ORDER BY startYear DESC;
```

### ROUND(значення, десяткові_знаки)

```sql
SELECT ROUND(AVG(averageRating), 2) FROM title_ratings;
```

### ORDER BY та LIMIT

```sql
SELECT primaryTitle, averageRating
FROM title_basics b
JOIN title_ratings r ON b.tconst = r.tconst
ORDER BY r.averageRating DESC
LIMIT 10;
```

### PRAGMA table_info() — Інспектування структури таблиці

SQLite-специфічне. Показує стовпці, типи та які стовпці є первинними ключами.

```sql
PRAGMA table_info(title_basics);
```

Стовпець `pk` у результаті - `0` для не-ключів та `1+` для стовпців PK.

## Нормалізація: формальні правила

Розділення грубої плоскої таблиці на менші, чистіші таблиці називається **нормалізацією**. Воно має три основні кроки, звані **нормальними формами**.

### "Ключ, весь ключ та ніщо, крім ключа"

**Перша нормальна форма (1NF) — "Ключ"**

Кожна комірка містить одне значення. Немає списків, немає розділених комами даних, немає багатозначних стовпців.

- *Порушення:* Таблиця гравців з коміркою `games_played`, що містить `"Rocket League, Valorant, Fortnite"`.
- *Виправлення:* Розбийте список на кілька рядків, щоб кожен рядок мав рівно одну гру.

**Друга нормальна форма (2NF) — "Весь ключ"**

Повинна бути в 1NF, І кожен стовпець, що не є ключем, залежить від **ВСЬОГО** первинного ключа (не тільки від частини композитного ключа). Це виправляє **часткові залежності**.

- *Порушення:* Таблиця з ключем (gamertag, game_name), де `player_email` залежить тільки від `gamertag`.
- *Виправлення:* Перемістіть `player_email` у власну таблицю Гравців з ключем `gamertag`.

**Третя нормальна форма (3NF) — "Ніщо, крім ключа"**

Повинна бути в 2NF, І кожен стовпець, що не є ключем, залежить **ТІЛЬКИ** від первинного ключа — не від іншого стовпця, що не є ключем. Це виправляє **транзитивні залежності**.

- *Порушення:* Таблиця гравців, де `coach_name` залежить від `team_name`, і `team_name` залежить від `gamertag`.
- *Виправлення:* Витягніть `team_name` та `coach_name` у таблицю Teams; залиште `team_name` у гравців як FK.

## Запитання для вивчення

**1. Що таке первинний ключ і чому він нам потрібен?**
Стовпець, який унікально ідентифікує кожен рядок. Без нього ви не можете надійно розрізнити записи або посилатися на них з інших таблиць.

**2. Чому імена зазвичай є поганими первинними ключами?**
Імена повторюються. Дві люди (або фільми, або продукти) можуть мати одне ім'я. Унікальний ID ніколи не робить.

**3. Як іноземний ключ створює зв'язок між таблицями?**
FK в одній таблиці містить значення, яке відповідає PK в іншій таблиці, пов'язуючи два рядки разом.

**4. Яка різниця між зв'язком 1:M та M:M?**
1:M використовує один FK на стороні "багатьох". M:M потребує з'єднавальної таблиці з двома FK.

**5. Які проблеми спричиняє надмірність?**
Трата місця, аномалії оновлення (ви пропускаєте деякі копії), аномалії вставлення (ви не можете додати часткові дані), та аномалії видалення (ви втрачаєте дані, які ви не мали на увазі видаляти).

**6. Як перевірити, чи правильне посилання іноземного ключа?**
Напишіть запит, який шукає дошлюзні рядки:
```sql
SELECT COUNT(*) FROM orders
WHERE customer_id NOT IN (SELECT customer_id FROM customers);
```
Він повинен повернути 0.

## Ключові висновки

1. Хороший дизайн бази даних стосується організації та узгодженості.
2. Надмірність — враг — кожен факт повинен зберігатися рівно один раз.
3. Первинні ключі ідентифікують рядки; іноземні ключи пов'язують таблиці.
4. Імена роблять погані первинні ключі — використовуйте унікальні ID.
5. Нормалізація (1NF → 2NF → 3NF) — це формальний процес для досягнення цього.
