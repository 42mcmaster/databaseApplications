Уявіть, що всі дані IMDb помішені в одну таблицю:

```
title      | year | rating | runtime | genre        | actor_name         | birth_year | category

Inception  | 2010 | 8.8    | 148     | Sci-Fi       | Joseph Gordon-Levitt| 1981      | actor
Inception  | 2010 | 8.8    | 148     | Sci-Fi       | Elliot Page        | 1987       | actor
Inception  | 2010 | 8.8    | 148     | Sci-Fi       | Tom Hardy          | 1977       | actor
```

## Чому надмірність — враг

**Надмірність (Redundancy) = Зберігання одного й того самого факту знову й знову**

**Трата місця в пам'яті** — "Inception" зберігається 19 разів замість одного разу

**Кошмари оновлення (Update Nightmares)** — Назва потребує виправлення? Оновіть 19 рядків. Пропустите один? Дані стають несумісними.

**Ризик помилки** — Хтось вводить "Incepton" в один рядок. Тепер пошуки ламаються.

## Рішення: розділити на окремі таблиці

**Фільми та ТБ-шоу** (`title_basics`):
```
ID фільму  | Назва     | Рік випуску | Жанр(и)
tt1375666  | Inception | 2010        | Action,Adventure,Sci-Fi
```

**Кредити акторів та знімальної групи** (`title_principals`):
```
ID фільму  | ID особи  | Тип ролі
tt1375666  | nm0000138 | actor        ← посилання на ID фільму
tt1375666  | nm1289434 | actor        ← той же фільм, інша особа
```

"Inception" зберігається **один раз**. Кожний кредит просто вказує на його ID.

## Первинні ключі (Primary Keys) — унікальні ID для кожного рядка

**Первинний ключ (Primary Key) = унікальний ідентифікатор для кожного рядка**
Як ваш номер учня, номер автомобіля або штрих-код.

**Правила:** Повинен бути унікальним, ніколи пустим (NULL), та стабільним.

| Таблиця | Первинний ключ | Приклад |
|---|---|---|
| Фільми (`title_basics`) | `tconst` (ID фільму) | tt1375666 = Inception |
| Люди (`name_basics`) | `nconst` (ID особи) | nm0000138 = Leonardo DiCaprio |

**Чому ID замість імен?** Два фільми називаються "The Batman." Два актори називаються "Chris Evans." ID ніколи не повторюються!

## Швидка перевірка

**Який первинний ключ краще для таблиці студентів?**

A) Повне ім'я студента
B) Email студента
C) Номер ID студента (наприклад, 12345)

**Відповідь: C** — ніколи не змінюється, гарантовано унікальний, короткий і простий.
Імена можуть повторюватися. Email можуть змінюватися. ID просто працюють.

## Іноземні ключи (Foreign Keys) — посилання між таблицями

**Іноземний ключ (Foreign Key) = стовпець, який вказує на первинний ключ іншої таблиці**

```
Фільми та ТБ-шоу (title_basics):
  ID фільму  | Назва
  tt1375666  | Inception

Кредити акторів та знімальної групи (title_principals):
  ID фільму  | ID особи  | Тип ролі
  tt1375666  | nm0000138 | actor      ← ID фільму вказує на таблицю Фільми
  tt1375666  | nm1289434 | actor      ← ID особи вказує на таблицю Люди
```

Зберігайте "Inception" **один раз**, посилайтеся на нього всюди.
Обидва `ID фільму` (tconst) та `ID особи` (nconst) — це іноземні ключі.

## Тест оновлення (Update Test)

**Поганий дизайн (одна велика таблиця):**
```sql
UPDATE big_table SET title = 'The Shawshank Redemption'
WHERE title = 'Shawshank Redemption';
-- Оновіть 15+ рядків (по одному на кожного актора/члена знімальної групи)!
```

**Хороший дизайн (окремі таблиці):**
```sql
UPDATE title_basics SET primaryTitle = 'The Shawshank Redemption'
WHERE tconst = 'tt0111161';
-- Оновіть 1 рядок!
```

Ми економимо **сотні тисяч дублікатів** у всій базі даних.

## Три ключі до хорошого дизайну

**1. Первинні ключи** — кожна таблиця потребує унікального ID (ID фільму, ID особи)

**2. Іноземні ключи** — пов'язуйте таблиці посиланнями, а не копіюючи дані

**3. Без дублікатів** — зберігайте кожен факт рівно один раз

## Введення до нормалізації

**Нормалізація (Normalization) = процес розділення однієї великої таблиці на менші, розумніші таблиці**

Це процес, через який пройшли дизайнери IMDb, щоб дійти до 5 чистих таблиць.

Ви будете робити те саме сьогодні — з новим набором даних.

---

## Питання нормалізації

Подивіться на будь-який рядок даних, що повторюються, і запитайте:

**"До якої сутності (Entity) ці дані належать?"**

```
Inception | 2010 | 8.8 | Leonardo DiCaprio | 1974 | actor
```

- `Inception, 2010, 8.8` — це **атрибути (Attributes)** сутності **Фільм**
- `Leonardo DiCaprio, 1974` — це **атрибути** сутності **Людина**
- `actor` — це описує **зв'язок (Relationship)** між ними

**Різні сутності → різні таблиці. Кожна сутність отримує свої атрибути.**

---

## Першa нормальна форма (1NF)

**1NF = атомні значення, первинні ключі, без повторюваних групп**

### Чотири основні правила 1NF

**1. Кожна комірка містить одне значення** — не списки, не розділені комами дані

**2. Немає змішування типів даних** — якщо стовпець цілих чисел, усі значення — цілі

**3. Кожна таблиця потребує первинного ключа** — унікальний ідентифікатор для кожного рядка

**4. Немає повторюваних груп** — один факт на рядок

### Приклад: вправа з інвентарем

**Поганий дизайн (порушення 1NF):**
```
| Гравець | Інвентар                           |
|---------|----------------------------------|
| trev73  | 3 стріли, 2 щити, 18 мідних монет  |
| jdog21  | 5 стріл, 2 амулети                 |
```

**Хороший дизайн (1NF):**
```
| Гравець | Тип предмета    | Кількість |
|---------|-----------------|-----------|
| trev73  | стріли          | 3         |
| trev73  | щити            | 2         |
| trev73  | мідні монети     | 18        |
| jdog21  | стріли          | 5         |
| jdog21  | амулети         | 2         |
```

---

## Друга нормальна форма (2NF)

**2NF = 1NF + кожен стовпець залежить від ВСЬОГО первинного ключа**

Це важливо, коли ваш ключ — **композитний (Composite Key)** (два або більше стовпців).

### Проблема з частковою залежністю

```
| Гравець | Тип предмета | Кількість | Рейтинг гравця |
|---------|-------------|-----------|-----------------|
| trev73  | стріли      | 3         | Advanced        |
| trev73  | щити        | 2         | Advanced        |
| trev73  | монети      | 18        | Advanced        |
| jdog21  | стріли      | 5         | Intermediate    |
```

Первинний ключ = **(Гравець, Тип предмета)**

`Рейтинг гравця` залежить **тільки від Гравця**, а не від цілого ключа. Це **часткова залежність (Partial Dependency)**, яка порушує 2NF.

### Рішення: розділити на дві таблиці

**Гравці:**
```
| Гравець | Рейтинг |
|---------|---------|
| trev73  | Advanced |
| jdog21  | Intermediate |
```

**Інвентар гравців:**
```
| Гравець | Тип предмета | Кількість |
|---------|-------------|-----------|
| trev73  | стріли      | 3         |
| trev73  | щити        | 2         |
```

---

## Третя нормальна форма (3NF)

**3NF = 2NF + жоден стовпець не залежить від іншого стовпця, що не є ключем**

Це виправляє **транзитивні залежності (Transitive Dependencies)**.

### Проблема з транзитивною залежністю

```
| Гравець | Рівень навички | Рейтинг гравця |
|---------|----------------|---|
| trev73  | 8              | Advanced |
| jdog21  | 5              | Intermediate |
| gila19  | 3              | Beginner |
```

`Рейтинг гравця` залежить від `Рівня навички`, а не безпосередньо від ключа. Це транзитивна залежність.

### Рішення: створити таблицю пошуку

**Гравці:**
```
| Гравець | Рівень навички |
|---------|---|
| trev73  | 8 |
| jdog21  | 5 |
```

**Рівні навички (довідник):**
```
| Рівень | Рейтинг |
|--------|---------|
| 1-3    | Beginner |
| 4-6    | Intermediate |
| 7-9    | Advanced |
```

---

## Золоте правило нормалізації

> **"Кожен атрибут у таблиці повинен залежати від ключа, від всього ключа та ніщо більше, окрім ключа."**

---

## Рецепт нормалізації

| Крок | Правило | Що виправляє |
|------|---------|--------------|
| **1NF** | Атомні значення, PK, без повторюваних груп | Таблиці, яких неможливо запитати |
| **2NF** | Кожен стовпець залежить від ВСЬОГО ключа | Часткові залежності |
| **3NF** | Кожен стовпець залежить ТІЛЬКИ від ключа | Транзитивні залежності |
